# Pix Space - 认证找回密码页面设计方案

## 元信息

- Status：draft
- 目标读者：产品、前端、测试
- 范围：新增认证域“找回密码”独立页面，支持邮箱提交与结果提示；不包含邮件模板、真实邮件发送链路、验证码与重置密码落地页
- 主要改动点：在 `views/auth` 下新增找回密码页面；在路由层新增匿名访问入口；登录页忘记密码入口改为跳转；补充对应 API、MSW mock 与最小测试覆盖
- Entry points：`src/views/auth/login/index.vue`、`src/views/auth/forgot-password/index.vue`、`src/router/index.ts`、`src/api/user.ts`、`src/mocks/handlers.ts`
- Acceptance checklist：按认证入口手工验收 + 关键路由与页面交互补充最小单测

## 1. 背景与目标

当前登录页中的“忘记密码？”仅为占位链接，未提供实际跳转与提交能力。随着视图目录已经按 `admin`、`portal` 分层，认证前页面仍保留在独立的 `auth` 域，因此本次“找回密码”能力应继续落在 `views/auth` 下，而不是混入 B 端或 C 端业务页面。

本次目标是补齐一个最小可用的找回密码流程：

- 用户从登录页点击“忘记密码？”进入独立页面
- 用户输入邮箱并提交
- 页面统一提示“如果该邮箱已注册，我们已发送重置链接”
- 不泄露邮箱是否存在

本次目标不追求真实邮件链路闭环，仅先完成前端页面、接口封装、Mock 行为与基础验证路径。

## 2. 设计原则

- **认证域独立**：找回密码属于认证前流程，统一放在 `views/auth` 体系，避免耦合到 `portal` 或 `admin`
- **体验一致**：背景与视觉氛围延续登录页，保证认证入口的一致性
- **最小闭环**：仅做邮箱提交型流程，不提前引入验证码、二次确认页或新密码输入页
- **安全优先**：对未知邮箱返回统一成功文案，避免枚举用户账号
- **分层清晰**：页面负责交互，API 负责请求，MSW 负责 mock，避免页面直接操作底层 request

## 3. 页面结构与交互

### 3.1 页面位置与路由

- 新增页面：`src/views/auth/forgot-password/index.vue`
- 新增路由：`/forgot-password`
- 路由属性：匿名可访问，`meta.requiresAuth = false`
- 登录页入口：将现有登录表单中的“忘记密码？”从空链接改为跳转 `/forgot-password`

选择 `auth` 而不是 `portal` / `admin` 的原因：

- 该页面发生在登录前，不属于任何业务端内页面
- 后续若增加 `reset-password`、`register`、`verify-email` 等能力，可继续归档到 `auth`
- 认证域页面更适合保持顶层匿名访问，不依赖业务布局

### 3.2 视觉与布局

- 延续登录页的星空背景与品牌文案氛围
- 页面主体仍采用居中卡片布局，保持与登录页一致的宽度级别与层级关系
- 卡片内容替换为：
  - 标题：`找回密码`
  - 说明文案：提示用户输入注册邮箱，我们会发送密码重置链接
  - 表单区域：邮箱输入框、提交按钮
  - 辅助入口：返回登录

推荐复用或参考现有登录页元素：

- `StarryBackground`
- 登录页容器层级与动画节奏
- 认证表单的输入框尺寸、主按钮尺寸与留白节奏

### 3.3 状态设计

页面需要明确以下状态：

- **初始态**：展示说明文案、邮箱输入框、发送按钮、返回登录入口
- **校验失败态**：邮箱为空或格式不合法时，在表单内直接提示错误
- **提交中**：按钮进入 loading，避免重复提交
- **提交成功态**：切换为成功提示区，文案统一为“如果该邮箱已注册，我们已发送重置链接，请注意查收邮箱”
- **提交失败态**：仅在请求异常或服务异常时提示失败信息，并保留重试入口

### 3.4 成功后的行为

提交成功后不自动跳转登录页，原因如下：

- 用户通常需要继续查看提示文案或切换邮箱重试
- 自动跳转会打断结果确认
- 当前阶段未接入真实邮件链路，停留在结果页更符合“已提交请求”的反馈节奏

因此成功态采用：

- 页面停留当前路由
- 显示统一成功文案
- 提供“返回登录”入口

## 4. 接口与数据约定

### 4.1 前端接口

在认证相关 API 中新增忘记密码提交方法，建议放在 `src/api/user.ts`，保持与当前登录接口同域管理。

建议接口形态：

```ts
type ForgotPasswordPayload = {
  email: string;
};

function forgotPassword(data: ForgotPasswordPayload): Promise<void>;
```

请求语义：

- 前端只提交邮箱
- 请求统一走 `request` 层
- 页面不直接操作 axios 或底层请求实例

### 4.2 Mock 行为

在 `src/mocks/handlers.ts` 中新增对应 handler，默认返回成功：

- 已知邮箱：返回成功
- 未知邮箱：同样返回成功
- 可选保留异常分支：用于测试网络失败或服务失败时的 UI 表现

统一成功返回的目的：

- 避免通过接口提示泄露账号存在性
- 与真实系统的安全策略保持一致

## 5. 实现拆解

### 5.1 页面层

- 新增 `src/views/auth/forgot-password/index.vue`
- 页面负责：
  - 表单状态
  - 提交状态切换
  - 成功/失败反馈展示
  - 返回登录导航

若页面交互保持简单，可维持单文件实现；若提交状态与行为分支增多，可新增页面私有 composable：

- `src/views/auth/forgot-password/useForgotPasswordSubmit.ts`

该 composable 仅服务当前页面，不提升到全局 `src/composables/`

### 5.2 登录页入口

需要调整现有登录表单组件的忘记密码入口，使其具备明确导航能力。建议保持组件职责简单：

- 由登录页或登录表单通过路由链接跳转 `/forgot-password`
- 不在登录页内内嵌忘记密码弹窗
- 不将忘记密码提交流程塞回 `LoginForm.vue`

### 5.3 路由层

在 `src/router/index.ts` 中新增顶层匿名路由：

- `path: '/forgot-password'`
- `name: 'ForgotPassword'`
- `meta.title: '找回密码'`
- `meta.requiresAuth: false`

守卫层无需新增新的权限模型，但需确认：

- 未登录访问 `/forgot-password` 不被拦截
- 已登录访问 `/forgot-password` 时按认证前页面统一策略重定向离开

本次建议：

- 与 `/login`、`/register` 保持一致：已登录用户访问 `/forgot-password` 时，重定向到 `sanitizeRedirectPath(to.query.redirect, '/explore')`

原因是：

- 认证前页面统一处理登录态，更利于维护和理解
- 已登录用户继续停留在找回密码页通常没有业务价值
- 可减少认证页访问策略分叉，避免后续路由守卫各写各的

## 6. 验收标准

- 登录页“忘记密码？”可跳转至 `/forgot-password`
- 找回密码页延续认证视觉风格，具备独立标题、说明、邮箱输入与发送按钮
- 邮箱为空或格式错误时，表单可阻止提交并展示校验提示
- 提交中按钮显示 loading，防止重复提交
- 提交成功后展示统一成功提示，不暴露邮箱是否存在
- 成功后页面停留当前页，且存在“返回登录”入口
- 请求异常时可展示错误提示并允许用户重试
- API、mock、路由、页面测试保持同步更新

## 7. 测试与验证

建议最小测试覆盖如下：

- `src/views/auth/login/index.test.ts`
  - 补充点击忘记密码入口后的跳转或链接行为验证
- `src/views/auth/forgot-password/index.test.ts`
  - 邮箱为空时不提交
  - 邮箱格式非法时不提交
  - 提交成功后显示统一成功态
  - 提交失败后展示错误提示
- `src/router/index.test.ts`
  - 匿名访问 `/forgot-password` 不被守卫拦截

手工验收路径：

- 访问 `/login`
- 点击“忘记密码？”
- 进入 `/forgot-password`
- 输入合法邮箱并提交
- 观察成功提示与返回登录入口
- 输入非法邮箱验证前端校验
- 模拟异常分支验证失败提示

## 8. 不做范围

以下内容明确不在本次实现范围内：

- 真实邮件发送能力
- 邮件模板配置
- 验证码输入流程
- 重置链接落地页
- 新密码设置页
- 多端不同找回密码表单
- 账号存在性差异化提示

## 9. Notes

当前方案以 Mock 成功链路为主，后续接真实后端时需要重点确认：

- 实际接口路径与返回结构
- 邮件发送限流策略
- 提交频控与防滥用策略
- 重置链接落地页的 token 校验方式
- 是否需要对已登录用户访问认证页做统一跳转策略（建议与 `/login`、`/register` 保持一致，统一重定向离开认证前页面）

后续如果进入真实重置闭环，建议在 `views/auth` 下继续补齐：

- `reset-password`
- `verify-email`（若需要）
- 更明确的认证域页面复用结构
