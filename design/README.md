# Pencil 原型协作与设计目录

将新增页面、或发生信息架构/交互大改页面的 Evolus Pencil 原型文件（`.pen`）提交到这里，作为页面评审与实现的事实来源。
本项目对这两类页面统一采用“先原型后落地”的方式，避免在 Vue 代码里反复试错。

设计规则与评审清单见 [AGENTS.md](AGENTS.md)；本文件只描述流程与目录协作方式。

适用范围：

- 新增页面
- 发生信息架构调整的页面
- 关键交互重做或需要先评审原型的页面

快速索引：

- 流程：见 [约定工作流](#1-约定工作流)
- 命名：见 [文件位置与命名](#2-文件位置与命名)

- 交接：见 [评审通过后的交接清单](#4-评审通过后的交接清单)
- 规则：见 [AGENTS.md](AGENTS.md)

## 1. 约定工作流

1. **需求沟通**：确认信息架构、功能范围、权限边界、关键状态（至少覆盖空/加载/错误；按页面类型补充权限或成功状态）
2. **原型设计**：等你显式确认"可以画图了"后，开始产出 Pencil 文件（`.pen`）
3. **原型评审**：以 Pencil 文件为准做评审与迭代，直到你确认"原型没问题"
4. **代码还原**：等你显式确认"可以将 pencil 文件还原为 vue 文件了"后，再开始将原型还原为 Vue 页面与组件

## 2. 文件位置与命名

- 存放目录：`design/`
- 默认命名格式：`<route>-<view>.pen`
  - `<route>`：去掉前导 `/`，多级路由用 `-` 连接
  - 动态路由参数替换为可读文字：`:userId` → `userid`，`:id` → `id`
  - `<view>`：用于补充页面用途（例如 `list`/`detail`/`edit`/`discover` 等）；若同一路由只有一个页面，可直接命名为 `<route>.pen`
  - `.pen` 文件名用于检索与追溯；文件内画板命名用于表达状态与交互（画板命名约定见 `design/AGENTS.md`）
  - 全部使用短横线命名；默认直接新增或修改原文件，不要求附加版本号
  - 仅当用户或评审过程明确要求“并行保留一版新方案”时，才新增 `-v<版本号>` 文件
  - 若已有历史 `-v<版本号>` 文件，可继续保留；后续若只是常规迭代，优先直接修改当前作为事实来源的那一份文件
  - 这里定义的是“.pen 文件名”规则；文件内每个画板名称仍遵循 `design/AGENTS.md` 中的 `<PageName>/状态` 约定
  - 示例：
    - `/admin/users` → `admin-users-list.pen`
    - `/explore` → `explore-discover.pen`
    - `/u/:userId/boards` → `u-userid-boards.pen`
    - `/board/:id` → `board-id.pen`
    - 明确保留多版时：`admin-users-list-v2.pen`

## 3. 评审交付物

评审阶段的唯一交付物是 `.pen` 文件（确保可以用 Evolus Pencil 打开查看）。
如需要在即时通讯/邮件中快速预览，可额外导出图片，并与 `.pen` 放在同目录下（导出物不作为唯一事实来源）。

最小样例建议：

- 1 个总览画板：用于表达页面主结构
- 至少 1 个关键状态画板：如空/加载/成功/失败
- 如有关键交互：增加交互后状态画板（如搜索结果、弹层、切换态）

## 4. 评审通过后的交接清单

- 当用户明确表示“可以将 pencil 文件还原为 vue 文件了”时，开发实现才允许开始。
- 进入开发前，至少确认以下信息：
  - `.pen` 文件路径
  - 评审结论（是否通过）
  - 待还原的画板范围
  - 当前仍未决的问题（如有）

## 5. 还原到 Vue 的范围边界

从 Pencil 还原到 Vue 代码时，原型主要服务于页面结构、状态与交互验收；具体工程分层仍以 `docs/FRONTEND_SPEC.md` 与现有目录规范为准。典型落地触点如下：

1. 路由与页面骨架（views + router）
2. 页面内拆分的可复用组件（components）
3. 请求与数据流（api + composables + stores）
4. 空/加载/错误等状态与交互细节（按原型对齐）
